# jsono_group_merge_min/max keyed aggregate optimization map

Goal: cut wall time and allocation pressure of `jsono_group_merge_min/max`
for a wide keyed-pair group-merge shape: a sticky shredded wide payload, a plain
detail payload, composite order keys, and groups with roughly single-digit to
tens of rows.

## Methodology

- Baseline and experiments use the same data, machine/session, and command shape.
- Main metric: `group_merge_keyed_max_jsono/100k/wide_shredded_keyed_many_groups`
  t8, because it is the closest in-repo proxy for the target shape: shredded wide
  values, composite key, many groups, JSONO-native output.
- Controls:
  - `group_merge_keyed_min_jsono/100k/wide_shredded_keyed_many_groups` t8,
    because `_min` uses the same executor with inverted sort-key direction.
  - `group_merge_keyed_max_jsono/100k/wide_flat_keyed_many_groups` t8, because
    plain input catches regressions outside the sticky-shredded finalize path.
  - `group_merge_keyed_max_jsono/100k/wide_shredded_keyed_few_groups` t8,
    because large groups stress accumulator growth.
- Parse, shredding, group/key materialization are outside timed SQL. The timed
  SQL measures only `GROUP BY` plus keyed aggregate execution.
- Experiments are cumulative only after a keep decision. Rejected code is removed
  before the next hypothesis.
- Complexity is recorded with `git diff --stat` and `git diff --numstat`, plus risk.

## Recommendation

Keep GB1R and GB1D. The mutable-tree rewrite with direct scalar leaf copies
clears the strict `_max` main metric threshold and improves every keyed control
measured in the same session. The follow-up destructive-combine patch improves
the wide keyed-pair proxy on both `threads=8` branches after the benchmark data
layout was corrected to use explicit Parquet row groups. The remaining check is
to rerun the private workload profile and decide separately whether a deeper
aggregate-state or projection change is still needed.

## Relation To Existing Ideas

- Existing keyed aggregate state stores two serialized JSONO blobs: merged
  values and a parallel key tree. Each row builds incoming value/key trees,
  serializes them, merges them with the accumulator, then serializes both
  accumulator blobs again.
- GB2 tests whether reusing recursive scratch vectors removes enough allocation
  overhead to matter without changing the state model.
- Rewrite tests the deeper model change: store a mutable tree whose leaves carry
  winning sort-key bytes and a standalone JSONO value, then serialize once in
  finalize.

### GB1R — mutable winner tree + direct scalar leaf blobs (KEEP)

- Patch: `src/jsono_merge.cpp` (+370/-304). Replaced
  `GroupMergeLWWState { values, keys }` with a heap-owned mutable tree:
  object nodes carry sorted JSON-key children; leaf nodes carry copied sort-key
  bytes and a standalone `OwnedJsonoBlob` value. Update walks each incoming
  `JsonoView` once, skips null/empty object members under IGNORE NULLS, updates
  only touched leaves, and fails loud on object-vs-leaf conflicts. Combine
  recursively merges source trees into target trees with deep copies for absent
  subtrees. Finalize serializes the tree once to plain JSONO and then uses the
  existing `JsonoShredFromSpec` path for sticky shredded returns.
- Root-cause follow-up inside the rewrite: the first mutable-tree version still
  missed the strict `_max` threshold because monotonic max keys make every row win
  and therefore serialize every scalar leaf. The kept version copies scalar leaf
  blobs directly into `OwnedJsonoBlob` instead of routing each scalar through
  `JsonoBuilder`; arrays still use the existing verbatim builder path.

### GB1D — destructive combine for keyed mutable trees (KEEP)

- Patch: `src/jsono_merge.cpp` (+51/-0 inside the cumulative diff). Added a
  destructive merge path for `JsonoGroupMergeLWWCombine` when DuckDB passes
  `AggregateCombineType::ALLOW_DESTRUCTIVE`. If the target state is empty, it
  takes ownership of the source tree. If both states have input, it recursively
  moves absent source children into the target and moves leaf payloads when the
  source leaf wins. Non-destructive combine retains the existing deep-copy path.
- Evidence source: native samples on the 12-row-group diagnostic file showed
  keyed aggregate samples concentrated in combine clone/merge work:
  `MergeLWWTreeIntoTree` and `CloneLWWTreeNode` dominated jsono frames. That
  file was not valid for parallel keep/reject, but the stack shape justified the
  small combine experiment.
- Benchmark correction: before evaluating GB1D, `keyed_pair` generation was
  fixed to write explicit `ROW_GROUP_SIZE 32768`. The regenerated 1M-row file has
  32 Parquet row groups. The old 12-row-group timings are ignored.
